# Jessie Work In `theories/jessie/`

The main result in this directory is a source-linked robust-safety theorem for
the Jessie `makeCounter` example:

- `make_counter.v` parses the Jessie `makeCounter` source into `JessicaAst`.
- The parsed `makeCounter` AST lowers directly to a HeapLang AST through
  `jessica_to_hla.v`.
- `make_counter.v` also parses and lowers a Jessie `checkedCounter` client.
- `checked_counter_from_source_safe` proves robust safety for a source-linked
  client that keeps the lowered `checkedCounter` shape and links its free
  `makeCounter` binding to the proved HeapLang `make_counter` constructor.

That theorem is the current bridge from source-facing Jessie syntax to the OCPL
robust-safety proof line for the upward-capability counter.

## Directory Layers

1. Jessie notation and PEG-backed parser path
- `jessie_notation.v`: Jessie-flavored object/string/update notation on top of
  HeapLang values
- `vendor/peg-coq/theories/`: minimal vendored `peg-coq` slice needed by the current parser, kept in the upstream-style `Peg` namespace so the copied theories can stay close to verbatim
- local delta from upstream is intended to stay minimal and explicit; at present the main adaptation is the Coq 8.9 compatibility patch in `vendor/peg-coq/theories/Match.v`
- `jessica_ast.v`
- `jessica_to_hla.v`: current partial `JessicaAst -> HeapLang AST` lowering, broad enough for the `makeCounter` path
- `quasi_json.v`
- `quasi_justin.v`
- `quasi_jessie.v`

2. Source generation
- `sources/makeCounter.js`
- `sources/escrow2013.js`
- `tools/js_to_coq_source.py`
- generated `makeCounter_js.v` and `escrow2013_js.v` (ignored in Git and regenerated by `make`)

3. Example targets and proofs
- `make_counter.v`: parser/lowering facts for `makeCounter` and `checkedCounter`, the hand-written proof-oriented counter theorem, and the source-linked `checked_counter_from_source_safe` theorem
- `escrow2013_target.v`: constructor-rich `JessicaAst` target for the current `escrow2013` rendition, aligned with the in-tree `.jessie.json` and `.jessie.lisp`
- `escrow2013.v`: exact-source recognition for `escrow2013_source`

## Proven Artifacts

- `parse_makeCounter_source_program`: `makeCounter_source` parses to the
  constructor-rich `makeCounter_jessica_program`.
- `jessica_to_hla_makeCounter_is_make_counter_binding`: the `makeCounter`
  Jessica module lowers to a HeapLang binding for `makeCounter`.
- `parse_checkedCounter_source_program`: `checkedCounter_source` parses to the
  constructor-rich `checkedCounter_jessica_program`.
- `jessica_to_hla_checkedCounter_program`: the `checkedCounter` Jessica module
  lowers to `checkedCounter_lowered_expr`.
- `checked_counter_safe`: robust safety for the proof-oriented client term.
- `checked_counter_from_source_safe`: robust safety for the source-linked
  lowered checked-client shape.
- `parse_escrow2013_source_program`: exact-source PEG recognition for the
  `escrow2013` source against `escrow2013_program`.

## Remaining Gaps

- `checked_counter_from_source_safe` links the lowered checked-client shape
  manually in Coq. It is not a general module-linking theorem for parsed Jessie
  modules.
- There is no general compiler-correctness theorem for `JessicaAst -> HeapLang`.
  The current lowering facts are example-specific equalities proved by
  computation.
- `jessica_to_hla.v` is intentionally partial. It covers the current
  `makeCounter` fragment but does not yet cover constructs such as `JIf` that
  appear in `escrow2013`.
- `escrow2013` uses exact-source PEG recognition through `exact_module_source`;
  it is not parsed by the structured Jessie grammar.
- `JThrow` needs semantics appropriate for `escrow2013` and promise behavior;
  treating it as simple stuckness would be misleading.

## Build

- Use the same switch the OCPL proof line uses:
  `eval "$(opam env --switch=ocpl-coq-8.9.1-ocaml-4.07.1)"`
- Then build from `packages/zoe/spec/_research/ocpl-coq/` with `make`.
- The top-level `Makefile` generates `theories/jessie/*_js.v` before running `coq_makefile`.
- `_CoqProject` lists those generated modules directly, so a clean checkout should start from top-level `make`, not from a direct `coq_makefile -f _CoqProject` or single-file `coqc` invocation.
